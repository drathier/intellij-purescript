# CPU Performance TODO — 2026-06-21

## Problem
IntelliJ PureScript plugin consumes ~256% CPU on idle with PureScript files visible.
Drops to 0 when no PS files are open. `purs ide server` process is idle.
Small project (14 files, 2690 LOC, 3 packages: .spago/g, .spago/p, BuildInfo).

### Evidence
- **Thread dump** at `/tmp/intellij-cpu.txt` (307169s elapsed = ~85h uptime)
  - Most threads in TIMED_WAITING during the snapshot (not actively burning CPU at that instant)
  - `BaseDataReader: output/error stream of purs` threads (6 instances, 3 generations) have ~3M ms cumulative CPU — but this is spread over 85h, average ~1% each
  - `Reference Handler` (516s CPU, 0.17% avg) is RUNNABLE in native `waitForReferencePendingList` — normal, not pathological
  - AWT-AppKit (6,899s CPU), Java2D Queue Flusher (3,600s), EDT (7,009s) — significant but from rendering, not plugin logic
- **IntelliJ Activity Monitor** data:
  ```
  316,1  <Process total CPU usage>
  256,0  Plugin Purescript: org.purescript.module
  ```
- **macOS Activity Monitor** also shows 200-300% for the Java process

### Key Facts
- CPU drops to 0 when no PureScript files are open (even with same project open, clicking in a JS file)
- `purs ide server` subprocess is completely idle (under 0.1% CPU)
- Disabling all PureScript-specific inspections reduces CPU ~50% (256% → ~128%)
- Scrolling reveals a 1-2 second delay for semantic highlighting: identifiers start white, then turn blue/green after 1-2s
- Project is small enough that any O(n) operation should be instant

### Architecture of Idle CPU Consumption
1. IntelliJ daemon highlighting pass fires when PS files are visible
2. Pass runs: (a) lexer → (b) `PSSyntaxHighlightAnnotator` → (c) all local inspections → (d) inlay hints → (e) external annotator
3. Each step above calls plugin code → IntelliJ attributes CPU to plugin
4. On idle, the pass should run once and stop — but if PSI reparse occurs (e.g., tree eviction from GC), a new pass re-triggers
5. The 256% suggests either: (a) very long passes, or (b) frequent re-triggering

## 1. (P0) Cache resolve() results per-element

Every `reference.resolve()` call recomputes from scratch:
traverses imports, calls `exportedValue(name)`, recursively walks re-exports.

**Approach**: Add a per-file result cache (e.g. `PsiElement → PsiElement?` map on `PSFile`).
Invalidate on `subtreeChanged()`.
This makes repeated resolve calls O(1) within a single pass.

**Files**:
- `src/main/kotlin/org/purescript/file/PSFile.kt` — add cache map + invalidation in `subtreeChanged()`
- All reference classes (`ExpressionIdentifierReference`, `TypeConstructorReference`, `ConstructorReference`, etc.) — check cache before full resolve

## 2. (P0) Don't call resolve() per-usage in the syntax annotator for highlighting

Two-tier resolution: cheap local-scope walk first, cached per-name resolve second.
Cuts resolve calls from ~500 to ~80 per pass.

## 3. (P1) Make inferType() incremental or cache results per-declaration

`SignatureInlayHintProvider.kt:84` calls `element.inferType()` on every signatureless ValueDeclarationGroup.
`PSFile.subtreeChanged()` resets `TypeSpace` entirely, so all inference is recomputed after every PSI change.

**Approach**: Keep `TypeSpace` across passes; only invalidate entries for changed subtrees
(use PSI element identity, not timestamp). Or cache `inferType()` results per declaration group.

**Files**:
- `src/main/kotlin/org/purescript/inference/TypeSpace.kt`
- `src/main/kotlin/org/purescript/inference/Inference.kt`
- `src/main/kotlin/org/purescript/module/declaration/SignatureInlayHintProvider.kt`

---

## 4. (P2) Refine subtreeChanged() granularity

`Module.kt:149` resets the entire `Cache` on every `subtreeChanged()`.
`PSFile.kt:74` resets the entire `TypeSpace` on every `subtreeChanged()`.
This means every small PSI change (even just re-parsing) triggers full recomputation of all lazy caches.

**Approach**: Instead of `cache = Cache()`, use finer-grained invalidation:
- Track which cache entries are dirty (e.g. `dirtyFlags` bitmask)
- Only recompute on access if dirty
- Or keep multiple cache generations and fall through to recomputation selectively

**Files**:
- `src/main/kotlin/org/purescript/module/Module.kt`
- `src/main/kotlin/org/purescript/file/PSFile.kt`

---

## 5. (P3) Make TypeConstructorReference.allCandidatesWithoutAlias lazy

`TypeConstructorReference.kt:53-66` eagerly builds a full list of ALL types
from the module + all imports before checking the element name.

**Approach**: Use sequence/`asSequence()` and `firstOrNull` so iteration stops
when a match is found. This avoids materializing the full candidate list.

**Files**:
- `src/main/kotlin/org/purescript/module/declaration/type/type/TypeConstructorReference.kt`
